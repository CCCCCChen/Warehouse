## Proposed Changes（按你的优先级拆分）

### Phase A：先完成“物品管理默认字段”调整（含折叠展示）

#### A1. 数据模型与 API（后端）

目标：先把字段打通，保持向后兼容（旧字段仍可用），并为 UI 分组/折叠提供完整字段。

1) 扩展 Item 表字段（SQLite 轻量迁移继续沿用 `ensure_items_table_columns()`）：
- **编码/标识**
  - `code`：TEXT，编码（按“类型+日期+序号”生成）
- **类型（两级）**
  - `type_l1`：TEXT（大类）
  - `type_l2`：TEXT（子类）
  - 兼容：保留现有 `category`（可作为旧数据展示/过滤的 fallback）
- **核心信息**
  - `usage`：TEXT（用途，长文本）
  - `image_path`：TEXT（图片保存路径或 URL）
- **时间信息**
  - `production_date`：DATE（生产日期）
  - `recorded_at`：DATETIME（记录日期，自动）
  - 保留现有 `purchase_date`：DATE（购买日期）
  - 保留现有 `expiry_date`：DATE
- **空间信息（先做文字级联）**
  - `room`：TEXT（房间/区域：玄关/厨房/客厅/过道/厕所/房间1..n/阳台…）
  - `spot`：TEXT（区域内位置：柜子/抽屉/墙面…，先文字）
  - 兼容：保留现有 `location`（可作为 fallback 或拼接后的展示字段）
- **状态属性**
  - `usage_status`：TEXT（在用/备用(囤货)/待维修/待处理）
  - `ownership`：TEXT（自有/借用）
- **财务价值（可空）**
  - `price`：REAL（购买价格）
  - `value_score`：REAL（使用价值/价值评分，先按数值存）
  - `replacement_cycle_days`：INTEGER（建议更换周期，天）
- **动态维度**
  - `usage_frequency`：TEXT（默认枚举：高/中/低/很少；允许自定义输入）
  - `related_item_ids`：TEXT（逗号分隔 id，或 JSON 数组字符串；第一版用逗号分隔）
  - `responsible_person`：TEXT（责任人）
- **自定义属性**
  - `custom_json`：TEXT（JSON 字符串，key/value）

2) 生成编码规则（类型+日期+序号）：
- 新增“每家庭每日期每大类/子类的序号”生成策略
  - 简化实现：在创建 item 时，按 `type_l1/type_l2 + recorded_at(YYYYMMDD)` 查询当日同类型数量，`count+1` 生成 `NNN` 序号
  - 编码格式示例：`家电-大家电-20260513-001`
  - 说明：该方案在高并发下可能存在竞态，但家庭场景/2C2G/SQLite 下可接受；如需严格唯一可再加唯一索引或事务锁（后续再做）

3) Schema/CRUD：
- 扩展 [schemas.py](file:///e:/PersonalFiles/Coding/Warehouse/app/schemas.py)
  - `ItemBase/ItemCreate/ItemUpdate/Item` 新增上述字段
  - `recorded_at` 在返回中给前端展示；创建时后端自动写入
- 更新 [models.py](file:///e:/PersonalFiles/Coding/Warehouse/app/models.py) 的 Item Column
- 更新 [main.py](file:///e:/PersonalFiles/Coding/Warehouse/app/main.py) 的 `ensure_items_table_columns()` wanted dict，补列 + 对旧行填默认值（如 recorded_at 为空则可留空）

4) 图片上传（第一版“支持上传图片”）：
- 新增接口（建议）：
  - `POST /api/items/upload_image`（multipart file）→ 返回 `{ image_url }`
  - 存储：`./data/uploads/{household_id}/{uuid}.{ext}`（目录已在 compose 文档里映射 `./data:/app/data`）
  - 限制：文件大小（如 5MB），只接受常见图片格式
  - 服务静态：FastAPI 挂载 StaticFiles，将 `/uploads/` 映射到 `data/uploads`
- Item 中 `image_path` 存储可直接用于 `<img :src="...">`

#### A3. 位置（先文字实现）+ 后续地图方案（只输出思路，不立即落地）

第一版（立即实现）：
- `room`：默认选项固定（玄关、厨房、客厅、过道、厕所、房间、阳台；允许用户自己输入“房间1/房间2”）
- `spot`：默认给几个常用（柜子/抽屉/墙面/台面/床底/冰箱），也允许自由输入
- 保存：room/spot 两字段；展示时可组合为 `room-spot`

后续“区域地图 + 墙面画框”方案（本计划只写思路，待你确认后再做测试界面）：
- 数据结构（建议存在 household_config 的新字段里）：rooms/areas/walls 的 JSON
  - room：矩形集合（可拼接/重叠），由用户命名
  - wall：在 room 下，按 东南西北 + 底面（可选）配置，wall 可关联一张照片 + 多个矩形框 + 名称
- 交互：全部在浏览器端完成画框（Canvas/SVG），后端只存 JSON 和图片文件
- 性能评估（2C2G）：
  - 服务器压力主要来自“图片上传/保存/静态分发”，不做服务端图像处理即可很轻
  - 建议限制图片尺寸（前端上传前压缩/限制分辨率可选，后续再加），限制上传大小
  - 画框计算在前端，后端只做读写，CPU/内存占用可控
